# Agent Rules & Instructions

이 프로젝트에서 작업하는 AI 에이전트들은 다음 규칙을 준수해야 합니다.

## 📝 Git 커밋 메시지 규칙 (Commit Message Rules)

커밋을 생성할 때는 기존 커밋 로그 양식을 분석한 아래 규칙을 엄격하게 준수하여 작성해야 합니다.

* **형식**: `<type>: <description>` (콜론 뒤에 공백 1칸을 둡니다.)
* **타입 (Type)**: 소문자로 작성합니다.
  * `feat`: 새로운 기능 추가
  * `fix`: 버그 수정, UI 수정, 오타 수정 및 세부 조정
  * `add`: 새로운 파일이나 리소스 추가
  * `refactor`: 코드 리팩토링
  * `chore`: 패키지 매니저 설정, 빌드 작업 등 단순 관리 작업
* **본문 (Description)**:
  * **한국어**로 작성합니다.
  * 문장의 끝에 마침표(`.`)를 찍지 않습니다.
  * 변경 사항을 명확하고 직관적으로 설명합니다.

### 💡 작성 예시
* `feat: 최신설교영상 기능 추가`
* `fix: 동영상 카드 UI 수정`
* `add: 유튜브 재생목록 추가(주일 오후 예배, 수요예배, 새벽예배)`
* `fix: 푸터 더미데이터 삭제 및 정보 최신화`


## ✅ Chrome 임시 프로필 실행 성공 기록

다음 명령으로 임시 프로필을 사용한 Chrome을 실행하고 `chrome-devtools-mcp`와 자동 연결했습니다.

```bash
killall "Google Chrome"
open -na "Google Chrome" --args \
  --remote-debugging-port=9222 \
  --user-data-dir=/tmp/chrome-remote-debug \
  --no-first-run &
npm exec chrome-devtools-mcp@latest --autoConnect
```

- **임시 프로필**: `/tmp/chrome-remote-debug` (북마크·로그인 정보 없음)
- **원격 디버깅 포트**: `9222`
- **MCP 연결**: 백그라운드 작업으로 자동 연결됨, 이후 `list_pages` 등 MCP 명령 사용 가능

앞으로 이 절차를 그대로 재사용하여 Chrome 디버깅 환경을 빠르게 구축할 수 있습니다.

## 🛠️ 하네스 엔지니어링 (Harness Engineering) 프로세스

이 프로젝트는 완성도 높은 개발을 위해 **Planer (설계자)**, **Generator (생성자)**, **Evaluator (평가자)** 3가지 역할의 에이전트 협업 구조를 적용합니다.

### 1. 에이전트별 역할 및 책임
* **Planer (설계자)**
  * **목적**: 코딩 작업 없이 오직 계획 및 구조 설계에만 집중합니다. (PRD 작성 및 구조화)
  * **산출물**: PRD(제품 요구사항 정의서), 기능 명세서, 컴포넌트 설계, 데이터 흐름 정의.
* **Generator (생성자)**
  * **목적**: 설계서(PRD)를 바탕으로 실제 고품질 코드를 작성합니다.
  * **산출물**: 완성된 소스 코드 및 Git 커밋.
* **Evaluator (평가자)**
  * **목적**: 작성된 코드가 설계서의 요구사항을 충족하는지, 버그나 스타일 이슈가 없는지 다각도로 평가합니다.

### 🚀 차기 진행 계획 (수정)
* **[완료] Planer 단계 (상세 설계)**: 실제 백엔드 소스 분석에 기초한 구체적인 데이터 규격 및 컴포넌트 설계 확정.
* **[대기] Generator 단계**: 설계서 내용을 바탕으로 코드 수정 작업 시작.
* **[대기] Evaluator 단계**: 아래 정의된 평가 목록 및 기준을 기반으로 크롬 브라우저 접속 테스트 및 검증 진행.

### 🛠️ 2. 평가자 수행 지침
1. **환경 로드**: 테스트 시작 전 `chrome-devtools-mcp`를 통해 `http://localhost:5173` 페이지에 접속하고 콘솔 에러가 없는지 1차 확인합니다.
2. **콘솔 로그 모니터링**: 각 기능(조회, 작성, 모달 실행 등) 인터랙션을 트리거할 때 프론트/백엔드 통신 에러 및 React warning이 없는지 실시간 확인합니다.
3. **피드백 루프**: 하나라도 통과 기준(Fail)에 못 미치면, 구체적인 에러 메시지나 화면 캡처 로그를 정리하여 **Generator**에게 수정을 요청합니다. 모든 기준이 통과되면 작업을 완료 처리합니다.

# 📄 [Planer 단계] 제품 요구사항 정의서 (PRD)
**최종 업데이트**: 2026-06-13  
**작성자**: Planer Agent  

---

## 1. 문서 개요 및 목적
본 문서는 성림교회 웹사이트의 미완성된 핵심 기능들(유튜브 설교 연동, 교회 앨범, 자유게시판, 사이트맵 모달)에 대한 제품 요구사항을 정의합니다. 백엔드에서 제공되는 실제 API와 데이터 모델을 기반으로 프론트엔드 컴포넌트와 화면 설계를 세분화하여, 사용자에게 버그 없는 매끄러운 UX를 선사하는 것을 목적으로 합니다.

---

## 2. 사용자 및 Use Case
* **일반 성도 및 방문자**:
  * 메인 페이지 및 예배 페이지에서 최신 설교 영상을 끊김 없이 시청할 수 있어야 합니다.
  * 교회 소식 및 앨범 탭에서 교회의 최근 사진들을 고화질로 모아볼 수 있어야 합니다.
  * 자유게시판에 글을 쓰고 지우며 다른 성도들과 원활하게 소통할 수 있어야 합니다.
  * 사이트맵을 통해 모바일/데스크톱 환경 전체 메뉴를 한눈에 파악하고 원하는 페이지로 바로 이동할 수 있어야 합니다.

| **UI/UX 구현** | 1. **방명록/피드형 카드 보드**: 전통적인 텍스트 테이블 대신, 각 메모가 포스트잇이나 세련된 카드 형태로 나열되는 Modern Card Board 디자인을 채택하여 rich한 비주얼을 제공합니다.<br>2. **글쓰기 입력 폼**: 폼은 컴팩트하고 유려하게 구성하며, 전송 시 부드러운 스피너와 함께 리스트에 슬라이드인 애니메이션으로 새 카드가 추가됩니다.<br>3. **삭제 액션**: 마우스를 호버할 때만 쓰레기통 아이콘이 나타나도록 하여 깔끔한 미니멀리즘을 유지합니다. |
| **예외 처리** | 1. 빈 내용이나 공백 문자만 입력 후 전송 시 "내용을 입력해 주세요." 유효성 검사 얼럿을 출력합니다.<br>2. 글자수 제한(예: 300자)을 걸어 데이터베이스 오버플로우나 레이아웃이 무너지는 현상을 원천 방지합니다. |
